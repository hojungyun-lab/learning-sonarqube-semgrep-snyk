# 🩹 07. Snyk 알려진 취약점(CVE) 완화 및 예외 처리 (Remediation)

## 📌 학습 목표 (Goal)
- 스캔된 취약점에 부여되는 `CVSS 점수(치명도)`의 의미를 깨닫습니다.
- 취약점을 1차적으로 수선하는 권장 패치 기법(Version Bumping)을 데모 앱에 적용합니다.
- 패키지 버전 상향이 불가능한 현업의 블로커(Blocker) 상황을 모방하여, Snyk CLI를 통해 임시 예외 처리(`.snyk` ignore)를 강제하는 방법을 배웁니다.

## 💡 핵심 딥다이브 (Deep-Dive)

### 1. CVSS 스코어와 무작정 패치가 답이 아닌 이유
취약점은 항상 국제 표준 점수인 **CVSS (Common Vulnerability Scoring System)** 가 매겨져 있습니다. (0.0 ~ 10.0점)

*   `Low / Medium (0.1 ~ 6.9)`: 권한이 있는 내부자만 접근할 수 있는 등 공격 난이도가 극도로 높은 경우.
*   `High / Critical (7.0 ~ 10.0)`: 그냥 URL 뒤에 변수 몇 개 붙여서 엔터만 쳐도 남의 서버를 털어먹을 수 있는 최고 등급. (Remote Code Execution, DB 탈취 등)

**무작정 최신 버전으로 올리면 안 되나요?**
안 됩니다. 패키지를 최신(Latest)으로 무조건 펌핑시키면 **Breaking Changes(기존 문법 호환 단절)** 로 인해 잘 돌아가던 앱이 죽어버립니다. Snyk의 가장 큰 장점은 무작정 최신 버전을 알려주는 게 아니라, **"그 취약점 딱 하나만 고쳐진 최소한의 안전 버전 (Fixed in)"** 을 핀셋으로 가이드해 준다는 점입니다.

---

## 🛠 실습 코드 (Hands-on) : 취약점 패치 및 면제(예외) 적용

### Step 1: 이상적인 해결책 (버전 업그레이드)

터미널에 출력된 `Fixed in` 권장 사항을 참조하여, `requests` 와 `fastapi` 라이브러리를 안전한 버전으로 끌어올립니다.
(Poetry가 `pyproject.toml`과 `poetry.lock`을 자동으로 업데이트하고 충돌을 방지합니다)

```bash
# 구 버전에서 비교적 안전한 마이너 버전 영역으로 변경 
# (uvicorn 0.18.2는 의도적으로 그대로 둠 - 블로커 상황 연출)
poetry add fastapi@0.86.0 requests@2.28.2
```

### Step 2: 현업의 블로커 상황 경험 및 Snyk Ignore 처리

업그레이드가 끝났으니 다시 `snyk test`를 돌려 파티를 즐기려고 하지만, 여전히 터미널이 붉습니다.

*상황 부여: 팀장이 말합니다. "uvicorn은 우리 레거시 시스템 구조상 절대 버전을 당장 올릴 수 없어! 어떻게 할래?"*

방화벽 안쪽 깊숙한 내부 관리자망이라 외부 공격 위험이 전무하다는 전제하에, 우리는 이 `uvicorn` 패키지의 특정 취약점에 대해 **면죄부(Ignore)** 를 발급해 주기로 합니다.

#### 2-1. 취약점 ID 획득
`snyk test` 결과 중 uvicorn 패키지가 가진 취약점의 고유 식별자(ID)를 찾습니다.
대개 `SNYK-PYTHON-UVICORN-10522` 혹은 `CVE-XXXX-XXXX` 형태로 명시되어 있습니다.

#### 2-2. Snyk Ignore 정책 커맨드 실행
이 취약점 코드를 30일 동안 무시하되, 그 이유를 로그로 남겨 감사를 대비합니다.

```bash
# 본인의 터미널에 나온 취약점 ID로 변경하세요
snyk ignore --id=SNYK-PYTHON-UVICORN-10522 \
  --expiry=2024-12-31 \
  --reason="내부 어드민망 전용으로 외부 노출 포트가 없어 공격 벡터 성립 희박 (30일 뒤 아키텍처 재검토)"
```

#### 2-3. 무시 정책 파일 생성 확인
위 명령어를 치면 숨김 폴더 위치에 `.snyk` 라는 파일이 생성됩니다. 이 파일 안에는 우리가 방금 작성한 "면죄 조건"이 YAML 포맷으로 들어갑니다. **이 `.snyk` 파일은 반드시 Git에 Commit 해야 팀원들과 CI 서버에서도 해당 에러를 무시합니다.**

### Step 3: 최종 검증

```bash
# 무시 규칙(.snyk) 반영된 스캔 실행
snyk test
```
이제 터미널 하단에 `Tested XX dependencies for known vulnerabilities, no vulnerable paths found.` 와 함께 초록색 Success 문구가 뜨게 됩니다! 드디어 우리의 데모 앱이 (임시방편을 포함하여) 깨끗해졌습니다.
